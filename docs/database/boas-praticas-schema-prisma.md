# Boas práticas para o schema Prisma da Comunikapp

Este documento existe porque a auditoria de banco de dados de 2026-07 encontrou 5 tabelas nunca usadas, 1 vulnerabilidade crítica e 1 alta por falta de isolamento multi-tenant, ~46MB de schemas/clients Prisma órfãos commitados no repositório, e a causa raiz de um estouro de memória no deploy. As regras abaixo existem para que esses problemas não se repitam. Cada regra cita o caso real que a motivou.

Ver também `docs/premissas melhores praticas.md` para as premissas gerais de arquitetura do projeto.

## Antes de criar uma tabela nova

- **Nome e propósito documentados.** Se a tabela nasce junto com uma feature que pode não ser concluída, deixe isso registrado no PR/commit. A causa mais comum de tabela órfã aqui foi feature abandonada no meio do caminho (`modo_impressao`, `historico_custo_maquinas`, `historico_custo_funcoes`) — não é proibido começar e não terminar, mas a tabela morta precisa ser removida quando a feature é abandonada, não deixada no schema para sempre.
- **`loja_id` em toda tabela que pertence a uma loja.** A Comunikapp é multi-tenant por linha (um banco só, isolamento via `loja_id`). Toda tabela nova com dado de uma loja específica precisa do campo `loja_id` e, na maioria dos casos, um `@@index([loja_id])` ou composto (`@@index([loja_id, algum_outro_campo])`).
- **`@@index` em toda foreign key.** MySQL/InnoDB não indexa FK automaticamente — só cria índice se você declarar. Sem índice, a query que seria um lookup vira um full table scan assim que a tabela cresce. Antes de mergear, confira se todo campo usado em `@relation(fields: [...])` tem um `@@index` correspondente (isolado ou como primeira coluna de um composto).
- **Tipo `Json` nativo para dado estruturado, não `@db.Text`/`@db.LongText` com `JSON.stringify` manual.** O schema atual tem 22 campos guardando JSON serializado como texto (contra só 12 usos do tipo `Json` nativo) — isso impede filtrar por uma chave do JSON sem trazer a linha inteira e fazer parse na aplicação. Use `Json` desde o início, a menos que o campo seja texto livre de verdade (observações, descrições).
- **Política de `onDelete` explícita e consciente.** Hoje, apagar uma `loja` cascateia por até 3 níveis em 70% das tabelas do banco. Isso é aceitável para dados operacionais (orçamentos, OS, itens), mas pense duas vezes antes de usar `onDelete: Cascade` em dados financeiros/históricos — nesses casos, prefira soft delete (campo `ativo`/`excluido_em`, já usado em `orcamento` e `OrdemServico`) e deixe o cascade físico só para tabelas filhas 1:N que não têm valor fora do pai.
- **Sem campo "temporário".** Comentários como `// Campo para V2` sem prazo de remoção viram dívida permanente — o schema atual tem 31 desses. Se o campo é mesmo temporário, marque com uma data/PBI de remoção ou não o adicione ainda.

## Segurança

- **Nunca resolver um recurso multi-tenant só por `id`.** Se o dado pertence a uma loja, toda leitura/escrita por id deve filtrar também por `loja_id` (`findFirst({ where: { id, loja_id } })`), nunca `findUnique({ where: { id } })` isolado. A ausência dessa checagem foi a causa da falha de IDOR encontrada em quatro métodos de transição de Ordem de Serviço nesta auditoria — um usuário de uma loja conseguia mudar o status de OS de outra loja só sabendo o id.
- **Nunca devolver segredo/código de aprovação/token na resposta HTTP.** Se o dado existe para ser entregue por um canal específico (e-mail, SMS), ele deve ir só por esse canal — nunca no corpo da resposta da API, mesmo que o endpoint pareça "interno". O bug mais crítico desta auditoria era exatamente isso: um endpoint devolvia o código de aprovação de orçamento em texto puro.
- **Todo endpoint que recebe `@Body()` precisa de DTO tipado com `class-validator`.** `@Body() dto: any` desativa a `ValidationPipe` global silenciosamente — não há erro, só falta de proteção. Isso abre espaço para mass-assignment (o cliente manda um campo que o service não esperava e ele vai parar no Prisma).
- **Nunca gerar ID manualmente.** Todo model já tem `id String @id @default(cuid())` — deixe o Prisma gerar. `Math.random().toString(36)` não é seguro criptograficamente e tem espaço de colisão pequeno; foi encontrado sobrescrevendo o default seguro em um service desta base.
- **Nunca logar segredo em texto puro**, nem em `console.log` nem em `Logger`. Log vai para arquivo/agregador e é mais difícil de rotacionar que um campo de banco.

## Processo de migration

- **Nome descritivo**, no padrão `<contexto>_<o_que_muda>` (ex.: `drop_tabelas_mortas_sem_uso`, `add_indices_fk_faltantes`) — evite nomes genéricos tipo `update` ou `fix`.
- **Revise o SQL gerado antes de aplicar em produção.** `prisma migrate dev` e `prisma migrate diff --script` geram o SQL; leia antes de rodar `migrate deploy`, especialmente em `DROP TABLE`/`DROP COLUMN`.
- **Nunca edite uma migration já aplicada.** Se algo estiver errado, crie uma nova migration corrigindo — editar uma migration já rodada em qualquer ambiente quebra a tabela `_prisma_migrations` (o histórico deixa de bater com o hash do arquivo).
- **Faça backup (`mysqldump`) antes de qualquer `DROP`/`ALTER` destrutivo em produção.** Para tabelas candidatas a remoção, rode um `SELECT COUNT(*)` em produção primeiro — se não estiver vazia como esperado, pare e investigue antes de aplicar.
- **Mantenha `prisma/` livre de schema solto.** Um schema Prisma só deve existir no repositório se algum script em `package.json` ou algum processo de deploy o referencia. Esta auditoria removeu ~46MB e 1.472 linhas de schemas/clients gerados que não eram usados por nada — isso não é só desperdício de espaço, foi a causa direta de um estouro de memória no `prisma generate` do deploy.
- **`prisma generate` não deve rodar mais de uma vez por build sem heap ampliado.** Se o script de deploy chama `prisma generate` manualmente antes de `npm run build` (que já dispara `prebuild → prisma generate` de novo), garanta que as duas chamadas usem o mesmo `NODE_OPTIONS=--max-old-space-size=...`.
- **Nunca crie uma pasta de migration com data "de mentira".** O nome da pasta (`YYYYMMDDHHMMSS_nome`) determina a ORDEM em que o Prisma aplica as migrations — inclusive num banco vazio do zero (disaster recovery, ambiente novo, shadow database do `migrate dev`). Em 2026-07 encontramos 5 migrations criadas com o timestamp placeholder `20250101` ("1 de janeiro") que nunca foi corrigido para a data real — elas ordenavam antes até da migration `_init`, e um `prisma migrate deploy` num banco vazio falhava na primeira delas. Sempre deixe o Prisma gerar o timestamp (`prisma migrate dev --name ...`) em vez de criar a pasta manualmente com uma data arbitrária.
- **Nunca altere o schema fora do `prisma migrate`.** Scripts ad-hoc que rodam `ALTER TABLE`/`RENAME TABLE` direto via `$queryRaw` ou `db push` para "resolver rápido" (ex.: `scripts/renomear-tabelas-estoque.ts` renomeando `inventory_*` para `estoque_*`) tiram a tabela do histórico de migrations rastreado. O efeito só aparece meses depois, quando alguém tenta reconstruir o banco do zero e a tabela simplesmente não existe no ponto em que uma migration posterior tenta alterá-la. Se precisar de uma mudança estrutural rápida, gere uma migration mesmo assim — mesmo que "fora de ordem" no seu fluxo de trabalho, ela precisa existir no histórico.
